> Цель файла: зафиксировать, что именно нужно переделать в сайте-портфолио, чтобы он перестал выглядеть как «красивый, но непонятный техно-экран» и стал понятной презентацией проектов для потенциального клиента из малого бизнеса.

## 1. Главная проблема текущего варианта

Сайт получился визуально красивый, но сейчас он больше похож на технический dashboard для разработчика, чем на понятную презентацию для клиента.

Главные ошибки:

1. Титульный экран сломан: слишком крупный шрифт, текст выходит за границы блока.
2. Блок `Automation Stack` непонятен для клиента и не объясняет пользу.
3. Меню дублируется: есть верхнее меню и еще одно меню ниже.
4. Названия проектов слишком технические: `Route MVP` и `Client Digest Hub` не объясняют, что получит бизнес.
5. Интерфейс перегружен кнопками и декоративными элементами.
6. Есть «ложные кнопки»: элементы выглядят как кнопки, но ничего не делают.
7. Разделы «Обо мне» и «Контакты» разделены, хотя для портфолио лучше объединить.
8. Меню при прокрутке визуально смешивается с контентом.
9. Нужно сделать нормальный мобильный вид.
10. Нужно добавить открытие PDF-презентаций по фиксированным именам файлов.
11. Нужна русская и английская версия сайта.
12. Нужно сделать цветовую схему чуть ярче: больше синего градиента, меньше почти черного фона.

## 2. Мое честное решение по UX

Я бы не пытался «подлатать» текущий экран точечно. Тут лучше сделать небольшой рефакторинг структуры.

Почему:

- проблема не только в размере шрифта;
- проблема в логике страницы;
- сейчас экран пытается одновременно быть портфолио, dashboard, меню, презентацией и техно-витриной;
- для малого бизнеса это слишком абстрактно.

Правильная структура должна быть такой:

```text
Верхнее меню
    ├── Обзор
    ├── Оптимизатор маршрутов
    ├── AI-отчеты и PDF
    ├── Обо мне и контакты
    └── RU / EN

Контентная область
    └── Показывается только один выбранный смысловой раздел
```

Главное правило:

> Пользователь нажал пункт меню — на экране должен быть только выбранный смысловой блок, без второго меню, лишних карточек и декоративных кнопок.

## 3. Новые названия разделов

Текущие названия слишком технические. Нужно заменить их на понятные для клиента.

| Было | Стало на русском | Стало на английском |
|---|---|---|
| `Обзор` | `Обзор` | `Overview` |
| `Route MVP` | `Оптимизатор маршрутов` | `Route Optimizer` |
| `Client Digest Hub` | `AI-отчеты и PDF` | `AI Reports & PDFs` |
| `Обо мне` + `Контакты` | `Обо мне и контакты` | `About & Contacts` |

Важно:

- слово `MVP` убрать из основных кнопок;
- `MVP` можно оставить только внутри текста мелким пояснением: «тестовая версия / демонстрационный прототип»;
- потенциальному клиенту сначала нужно понять пользу, а не технический статус проекта.

## 4. Что убрать полностью

### 4.1. Убрать блок `Automation Stack`

Этот блок сейчас не нужен.

Почему:

- клиент не понимает, зачем ему `Automation Stack`;
- `Routes`, `Reports`, `AI`, `OpenRouter`, `CRM`, `PDF`, `Client link` выглядят как набор технических плиток;
- часть элементов выглядит как кнопки, но не является кнопками;
- блок не продает пользу.

Вместо него нужно сделать понятный блок:

```text
Что можно автоматизировать

1. Маршруты и выезды
2. Отчеты и PDF-документы
3. Уведомления и боты
4. Мини-CRM под конкретный бизнес
```

Но этот блок должен быть в разделе `Обзор`, а не отдельным dashboard-экраном.

---

### 4.2. Убрать второе меню внутри страницы

На скриншоте есть верхнее меню и еще одно меню ниже:

```text
Обзор | Route MVP | Client Digest Hub | Обо мне | Контакты
```

Нижнее меню нужно убрать.

Должно остаться только одно меню — верхнее.

---

### 4.3. Убрать «ложные кнопки»

Убрать или изменить элементы вроде:

```text
R
ROUTE MVP
Live MVP
PORTFOLIO PRESENTATION
```

Если элемент не нажимается, он не должен выглядеть как кнопка.

Допустимые варианты:

| Элемент | Как должен выглядеть |
|---|---|
| Логотип `R` | Небольшой квадратный знак без hover-эффекта кнопки. |
| `Portfolio presentation` | Текстовый бейдж без поведения кнопки. |
| `Route MVP` | Не использовать в основном UI. Заменить на `Оптимизатор маршрутов`. |
| `Live MVP` | Убрать или заменить на понятный статус: `Тестовая версия`. |

## 6. Требования к дизайну

## 6.1. Общий стиль

Оставить темный технологичный стиль, но сделать его ярче и понятнее.

Ориентир:

- фон темный, но не полностью черный;
- карточки с градиентом от ярко-синего к темно-синему;
- кнопки ярко-синие/голубые, как в примере интерфейса;
- меньше серых и почти черных блоков;
- больше визуального разделения между кнопками, карточками и обычными надписями.

### Цветовая палитра

Можно использовать примерно такие CSS-переменные:

```css
:root {

  --panel-2: #17294a;
```

```css
body {
  background:
    radial-gradient(circle at 20% 10%, rgba(32, 140, 255, 0.22), transparent 32%),
```

Кнопка должна выглядеть как кнопка:

## 7. Исправление титульного экрана

Сейчас главный заголовок слишком большой и выходит за рамки.

Нужно сделать адаптивный размер:

```css
.hero-title {
  font-size: clamp(38px, 6vw, 76px);
  line-height: 1.04;
  letter-spacing: -0.04em;
  max-width: 980px;
  overflow-wrap: break-word;
}
```

Для мобильного:

```css
@media (max-width: 640px) {
  .hero-title {
    font-size: clamp(34px, 11vw, 52px);
    line-height: 1.06;
  }
}
```

Важно:

## 15. Автоматическая публикация на GitHub Pages через кнопку

Самый простой вариант «кнопки» — файл `publish.cmd`, который запускает PowerShell-скрипт и делает `git add / commit / push`.

Это не кнопка на сайте, а кнопка-файл на твоем ПК: двойной клик — и изменения отправляются на GitHub.

## 15.1. Файл `publish.cmd`

Создать в корне проекта файл:

```text
publish.cmd
```

Содержимое:

```bat
@echo off
powershell -ExecutionPolicy Bypass -File "%~dp0publish.ps1"
pause
```

## 15.2. Файл `publish.ps1`

Создать в корне проекта файл:

```text
publish.ps1
```

Содержимое:

```powershell
Set-Location -Path $PSScriptRoot

Write-Host "Проверяю изменения..." -ForegroundColor Cyan
git status

$defaultMessage = "Update portfolio site"
$message = Read-Host "Введите комментарий к публикации или нажмите Enter"
if ([string]::IsNullOrWhiteSpace($message)) {
    $message = $defaultMessage
}

Write-Host "Добавляю файлы..." -ForegroundColor Cyan
git add .

Write-Host "Создаю commit..." -ForegroundColor Cyan
git commit -m $message

Write-Host "Отправляю на GitHub..." -ForegroundColor Cyan
git push

Write-Host "Готово. Через несколько минут GitHub Pages обновит сайт." -ForegroundColor Green
```

## 15.3. Что будет делать эта «кнопка»

1. Покажет изменения.
2. Спросит комментарий к публикации.
3. Выполнит `git add .`.
4. Выполнит `git commit -m "..."`.
5. Выполнит `git push`.
6. GitHub Pages сам обновит сайт.

Важно:

- если изменений нет, `git commit` может написать, что коммитить нечего;
- это не ошибка;
- если GitHub попросит авторизацию, нужно будет пройти ее один раз.

## 19. Справочник терминов и сокращений

| Термин | Простое объяснение |
|---|---|
| MVP | Минимальная рабочая версия продукта для проверки идеи. В клиентском интерфейсе лучше не использовать это слово как главное название. |
| GitHub Pages | Бесплатная публикация статического сайта из GitHub-репозитория. |
| HTML | Разметка страницы: заголовки, блоки, ссылки, кнопки. |
| CSS | Внешний вид сайта: цвета, размеры, сетки, адаптивность. |
| JavaScript / JS | Логика сайта: переключение разделов, языков, активных кнопок. |
| PDF | Файл-презентация или документ, который можно открыть и отправить клиенту. |
| CRM | Система учета клиентов, заявок, задач и статусов. |
| API | Способ соединить сайт с внешним сервисом: картами, почтой, AI, CRM, базой данных. |
| Telegram-бот | Программа, которая отправляет уведомления или принимает команды в Telegram. |
| GPS-маячок | Устройство или приложение, которое передает координаты объекта, машины или сотрудника. |
| LocalStorage | Память браузера, где можно сохранить выбранный язык или настройки сайта. |
| SMTP | Технический способ отправки e-mail. На бесплатных хостингах может быть ограничен. |
| Email API | Более надежный способ отправки писем через специальный сервис. |
