## 7. Ответ на вопрос: зачем мы запускаем мини-сервер на ПК

Команда:

```powershell
python -m http.server 8000
```

запускает маленький локальный сервер, который раздает файлы сайта из текущей папки.

### Без него сайт откроется?

Иногда да. Если сайт состоит только из простого `index.html`, `style.css` и `script.js`, его можно открыть двойным кликом по `index.html`.

Но это не лучший вариант.

### Почему лучше через `http://localhost:8000`

Причины:

1. **Так сайт ведет себя ближе к реальной публикации на GitHub Pages.**  
   На GitHub сайт открывается не как файл `file:///...`, а через обычный веб-адрес `https://...`.

2. **Лучше работают относительные пути.**  
   Например:

   ```text
   assets/presentations/client_digest_hub.pdf
   assets/images/screen.png
   script.js
   style.css
   ```

3. **Меньше проблем с JavaScript.**  
   Некоторые вещи в браузере могут блокироваться при открытии через `file:///`, особенно если сайт позже будет использовать `fetch`, JSON-файлы, модули или загрузку локальных данных.

4. **Проще проверять PDF и картинки.**  
   Если ты положишь PDF в папку `assets/presentations`, то через сервер можно проверить, что ссылка открывается именно так, как она будет открываться после публикации.

5. **Это безопасно для разработки.**  
   Сервер работает только на твоем компьютере для локальной проверки. Это не production-сервер и не публикация в интернет.

### Простыми словами

Мини-сервер нужен не потому, что сайт обязательно без него не откроется. Он нужен, чтобы сайт на твоем ПК работал почти так же, как потом будет работать на GitHub Pages.

## 8. Папка `manual` для .md-файлов с нашей беседой

Да, папку `manual` можно создать.

Например:

```text
D:\MyProgect\portfolio-site\manual
```

Туда можно складывать:

```text
manual\13-05-26_01_сайт_презентация_портфолио.md
manual\13-05-26_02_переделка_сайта_портфолио.md
manual\13-05-26_03_точечные_правки_сайта_портфолио.md
```

Но если ты не хочешь, чтобы эта папка попадала на GitHub, нужно добавить ее в `.gitignore`.

### Что сделать

В корне проекта:

```text
D:\MyProgect\portfolio-site\.gitignore
```

добавь строку:

```gitignore
manual/
```

Рекомендуемый `.gitignore` для твоего сайта:

```gitignore
# Личные инструкции, черновики и переписка
manual/

# Системные файлы Windows
Thumbs.db
Desktop.ini

# VS Code, если не хочешь публиковать настройки редактора
.vscode/

# Временные файлы
*.tmp
*.log
```

### Важный момент

Если папка `manual` уже была добавлена в Git раньше, одного `.gitignore` будет недостаточно.

Тогда нужно выполнить:

```powershell
git rm -r --cached manual
```

Эта команда уберет папку `manual` из контроля Git, но не удалит ее с компьютера.

После этого:

```powershell
git add .gitignore
git commit -m "Ignore manual notes"
git push
```

## 9. Как сделать кнопку-автоматизатор для загрузки на GitHub

Самый простой и надежный вариант на Windows — сделать файл `publish.cmd` в корне проекта. Его можно запускать двойным кликом как кнопку.

### Важно понимать

Настоящую кнопку внутри сайта, которая сама делает `git push`, делать нельзя и не нужно. Браузер не должен иметь доступ к твоей файловой системе и GitHub-командам. Это ограничение безопасности.

Правильная «кнопка» для твоего случая — локальный файл:

```text
D:\MyProgect\portfolio-site\publish.cmd
```

Ты нажимаешь на него двойным кликом, и он отправляет изменения на GitHub.

Если хочешь более контролируемый вариант, можно сделать PowerShell-скрипт.

$ErrorActionPreference = "Stop"

## 16. Что я считаю важным по дизайну

Сейчас главная проблема была не в том, что сайт некрасивый. Он как раз визуально уже неплохой. Проблема была в том, что разные элементы выглядели одинаково:

```text
кнопки = теги = подписи = элементы интерфейса
```

Из-за этого пользователь не понимает, куда нажимать и что является просто текстом.

Правильная логика должна быть такой:

| Тип элемента | Как должен выглядеть |
|---|---|
| Верхнее меню | Отдельная навигация, спокойно и компактно. |
| Action-кнопки | Яркие синие кнопки с градиентом и hover-эффектом. |
| Контакты | Кнопки, потому что по ним можно перейти. |
| Технологии | Обычная строка текста, без ощущения клика. |
| Информационные блоки | Большие красивые градиентные карточки с понятным текстом. |
| Заголовки | Крупные, но не гигантские; не должны вылезать за экран. |
