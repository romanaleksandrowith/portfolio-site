## 0. Как запускать локальный сервер после включения компьютера

> Этот блок нужен каждый раз, когда ты включил ПК и хочешь посмотреть сайт локально до публикации на GitHub Pages.

### Команды PowerShell

```powershell
cd "D:\MyProgect\portfolio-site"
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
python -m http.server 8000
```

### Что означает каждая команда

| Команда | Что делает |
|---|---|
| `cd "D:\MyProgect\portfolio-site"` | Переходит в папку проекта сайта-портфолио. Если папка у тебя называется иначе, нужно указать фактический путь. |
| `Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass` | Разрешает выполнение PowerShell-скриптов только в текущем окне PowerShell. После закрытия окна настройка сбросится. Это безопаснее, чем менять политику навсегда. |
| `python -m http.server 8000` | Запускает простой локальный веб-сервер для статического сайта. |

### Локальные адреса сервера

| Адрес | Что смотреть |
|---|---|
| `http://127.0.0.1:8000/` | Основная локальная версия сайта. |
| `http://localhost:8000/` | То же самое, альтернативный адрес. |

Проверять нужно: открывается ли главная страница, работают ли вкладки, корректно ли отображается мобильная версия, открываются ли ссылки на проекты, GitHub, e-mail и Telegram.

## 1. Главная цель сайта

Нужно сделать не просто страницу с двумя ссылками, а аккуратный сайт-презентацию, который показывает тебя как специалиста, способного:

- быстро собирать работающие MVP;
- превращать ручные процессы в цифровые сценарии;
- связывать аналитику, отчёты, маршруты, AI и бизнес-процессы;
- адаптировать техническую основу под конкретную задачу клиента.

Главный смысл сайта:

> **Эти проекты — не финальные коробочные продукты, а рабочие демонстрационные основы. На их базе можно развивать индивидуальные решения под логистику, отчётность, аналитику, CRM, автоматизацию уведомлений и AI-помощников для бизнеса.**

Важно: фразу «можно реализовать абсолютно любую идею» лучше не писать напрямую. Она звучит слишком широко и может вызвать недоверие. Лучше формулировать профессионально:

> **На базе этих MVP можно быстро проверять и развивать разные бизнес-сценарии: от маршрутизации и отчётности до специализированных рабочих кабинетов и внутренних CRM.**

## 2. Какой сайт делаем

### Формат

Делаем **одностраничный сайт-презентацию** с вкладками внутри одной страницы.

Это лучше, чем много отдельных HTML-страниц, потому что:

- быстрее сделать;
- проще поддерживать;
- удобно показывать как презентацию;
- хорошо подходит для GitHub Pages;
- можно менять текст и сразу публиковать обновления через `git push`.

### Рекомендуемая структура вкладок

| Вкладка | Назначение |
|---|---|
| `Обзор` | Главный экран: кто ты, какие проекты показываешь, чем полезен. |
| `Route MVP` | Отдельная вкладка проекта оптимизации маршрутов. |
| `Client Digest Hub` | Отдельная вкладка AI/PDF-дайджестов. |
| `Обо мне` | Краткое описание опыта, компетенций и направления работы. |
| `Контакты` | E-mail, Telegram, GitHub, ссылки на проекты и портфолио. |

## 3. Визуальный стиль

Ты приложил пример интерфейса в стиле тёмной финансовой панели: карточки, графики, тёмный фон, синий акцент, аккуратная панель навигации. Мы не копируем крипто-интерфейс буквально, но берём из него визуальную идею.

### Что берём из примера

- тёмный фон;
- мягкие синие и фиолетовые акценты;
- крупные карточки с закруглениями;
- ощущение dashboard / SaaS-продукта;
- верхняя панель навигации;
- блоки со статусами, метриками и возможностями;
- современный вид, без перегруза.

### Что не берём

- криптовалютную тематику;
- свечные графики как основной элемент;
- банковские карты и финансовые балансы;
- слишком мелкий текст.

### Итоговое направление дизайна

> **Тёмный SaaS-dashboard для портфолио: слева/сверху навигация, в центре карточки проектов, справа/ниже — показатели, стек технологий, сценарии развития и контакты.**

## 4. Рекомендуемый технический вариант

### Выбор технологии

Для первого этапа я предлагаю делать сайт как обычный статический frontend:

- `index.html`
- `style.css`
- `script.js`

Без React, Vite, сборщиков и npm.

### Почему так лучше сейчас

| Причина | Пояснение |
|---|---|
| Быстро | Можно собрать рабочий сайт за один подход. |
| Надёжно | GitHub Pages отлично подходит для HTML/CSS/JS. |
| Меньше ошибок | Не будет проблем со сборкой, зависимостями и деплоем. |
| Проще редактировать | Тексты можно менять прямо в HTML или JS. |
| Подходит для портфолио | Нам важна презентация, а не сложная frontend-архитектура. |

React/Vite можно подключить позже, если сайт начнёт расти: фильтры, много проектов, админка, автогенерация карточек из JSON и так далее.

## 8. Вкладка проекта `Client Digest Hub`

### Короткое позиционирование

```text
Client Digest Hub — AI-сервис, который превращает шаблон и короткое пожелание клиента в готовый HTML/PDF-дайджест. Пользователь не работает с prompt engineering напрямую: он выбирает сценарий и получает понятный результат.
```

### Что уже работает

```text
В демо уже есть тёмный диспетчер проектов, карточки клиентов, выбор шаблона, необязательные пожелания, генерация AI-дайджеста через OpenRouter, сохранение результата в HTML/PDF, клиентская ссылка RU/EN и деплой на Render через Docker.
```

### Блок функций

| Функция | Описание |
|---|---|
| Диспетчер проектов | Внутренний экран для создания проекта, выбора шаблона и управления клиентскими сценариями. |
| Клиентская ссылка | Клиент получает отдельную ссылку, где может выбрать шаблон и добавить пожелание. |
| AI-генерация | Система формирует текст результата на базе шаблона и уточнения пользователя. |
| HTML/PDF | Результат сохраняется в удобном виде, который можно открыть, скачать или отправить. |
| RU/EN | Клиентский сценарий можно развивать в двух языковых версиях. |
| Render/Docker | Проект уже прошёл путь от локальной разработки до публичного демо-деплоя. |

### Важное уточнение про e-mail

```text
Функционал отправки результата по e-mail заложен в проектную логику, но на бесплатном Render могут быть ограничения SMTP-портов и технические сложности с почтовой отправкой. Для production-версии лучше подключать отдельный email API. При необходимости такой сценарий можно также развернуть локально на ПК или на собственном сервере, где почтовые настройки контролируются отдельно.
```

### Как объяснять развитие проекта

```text
Client Digest Hub — это не только генератор коротких отчётов. Это основа для индивидуальных процессов: регулярных справок, клиентских порталов, анализа переписок, Telegram-чатов, открытых источников, внутренних документов и динамики показателей во времени.
```

### Примеры развития под бизнес

| Направление | Что можно сделать на базе Client Digest Hub |
|---|---|
| Персональные отчёты | Автоматически создавать PDF/HTML-сводки под конкретного клиента или руководителя. |
| Telegram-аналитика | Анализировать сообщения в Telegram-чатах и формировать сводку по темам, рискам и динамике. |
| Клиентский портал | Давать клиенту ссылку, где он выбирает шаблон результата и получает документ. |
| Внутренние справки | Формировать служебные записки, инструкции, отчёты и дайджесты по шаблонам. |
| Мониторинг открытых источников | Собирать новости, изменения, упоминания и превращать их в регулярный дайджест. |
| Аналитика динамики | Накапливать данные по дням/неделям и показывать изменения: рост обращений, частые проблемы, повторяющиеся темы. |
| Локальная версия | Развернуть систему на ПК или внутреннем сервере для сценариев, где данные нельзя отправлять во внешние сервисы. |

## 13. Автоматическая публикация на GitHub Pages

### Главная идея

GitHub Pages может публиковать сайт из репозитория. Для простого HTML/CSS/JS сайта лучший вариант:

- ветка: `main`;
- папка публикации: `/root`, то есть корень проекта;
- после `git push` сайт обновляется на GitHub Pages.

### Почему «кнопку на сайте» делать не надо

Обычная кнопка внутри сайта не может безопасно выполнить на твоём ПК команды:

```powershell
git add .
git commit
git push
```

Браузер специально запрещает сайтам запускать локальные команды на компьютере. Это защита от вредоносных сайтов.

Зато можно сделать удобную кнопку-файл на ПК:

- `publish.cmd` — запускается двойным кликом;
- внутри вызывает `publish.ps1`;
- скрипт сам делает `git add`, `commit`, `push`.

Это и будет практичный вариант «кнопки публикации».

---

## 14. Файл `publish.ps1`

Создать в корне проекта файл:

```text
D:\MyProgect\portfolio-site\publish.ps1
```

Содержимое:

```powershell
param(
    [string]$Message = "update portfolio site"
)

Write-Host "Переходим в папку проекта..." -ForegroundColor Cyan
Set-Location $PSScriptRoot

Write-Host "Проверяем изменения..." -ForegroundColor Cyan

Write-Host "Добавляем файлы в commit..." -ForegroundColor Cyan

Write-Host "Создаём commit..." -ForegroundColor Cyan

Write-Host "Отправляем изменения на GitHub..." -ForegroundColor Cyan
git push

Write-Host "Готово. Проверь сайт GitHub Pages." -ForegroundColor Green
```

## 19. Мой честный вывод по задаче

Идея хорошая и правильная для твоей текущей стадии.

Сейчас у тебя уже есть два демонстрационных проекта, но их нужно упаковать в один понятный входной экран. Для потенциального заказчика, работодателя или партнёра важно быстро понять:

1. кто ты;
2. что уже сделал;
3. какую бизнес-задачу это решает;
4. как это можно адаптировать под конкретную компанию;
5. как с тобой связаться.

Самая сильная подача:

> **Я не просто делаю отдельные скрипты. Я умею собрать работающий MVP вокруг бизнес-процесса: данные → логика → интерфейс → отчёт → автоматизация → развитие.**

Самый рискованный момент:

> Не надо обещать слишком много в стиле «можно сделать абсолютно всё». Лучше показывать конкретные направления: маршрутизация, отчётность, CRM, Telegram-боты, GPS-данные, AI-дайджесты, внутренние рабочие кабинеты.
